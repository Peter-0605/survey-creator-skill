# Performance benchmark notes

This directory archives empirical performance benchmark results for the current HTML runtime generated by `survey-creator-skill`.

## Benchmark date
- 2026-04-24

## Test goal
Estimate how far the current generated HTML runtime can scale before a questionnaire needs explicit performance optimization.

## Test model
The benchmark used generated surveys with these characteristics:

- `onePageOneQuestion = false`
- manual pagination via `Pagination`
- around 5 questions per page
- mixed question types:
  - `radio`
  - `checkbox`
  - `input`
- mixed logic actions:
  - `show_question`
  - `auto_select_option`
  - `hide_option`
- browser-based load and full interaction walk-through using Playwright

## Measured results

| Questions | Logic rules | Screens | HTML size | Initial load | Full browser walk-through |
|---|---:|---:|---:|---:|---:|
| 100 | 148 | 23 | 191.9 KB | 444 ms | 7.0 s |
| 200 | 298 | 43 | 322.6 KB | 507 ms | 38.1 s |
| 300 | 448 | 63 | large DOM (~939 KB outer HTML) | 608 ms | 116.0 s |
| 400 | 598 | not fully completed | larger | page loaded | full interaction exceeded 240 s timeout |

## Interpretation

### What stayed acceptable
- first-load behavior remained relatively stable through 300 questions
- the generated page could still open successfully even at large questionnaire size

### What degraded quickly
- full end-to-end interaction time increased sharply after 200 questions
- repeated page transitions + logic recomputation + DOM updates were the main cost drivers

## Practical guidance

### Comfortable zone
- up to ~100 questions
- up to ~150 logic rules

### Still usable but should be watched
- up to ~200 questions
- up to ~300 logic rules

### Optimization strongly recommended
- 300+ questions
- 400+ logic rules

## Conclusion for typical product usage
For normal production surveys, the current generated runtime is generally fine if the questionnaire stays below roughly 100 questions.

If your real scenario is:
- within 100 questions
- but logic rules are relatively dense

then the current runtime is still likely workable, but you should pay attention to:
- page grouping density
- how many logic rules depend on the same source questions
- how much rich text / media exists per page

## Suggested next optimization threshold
If a real survey regularly approaches either of these conditions, start performance optimization work:
- 100 questions + 200+ logic rules
- 150+ questions regardless of logic density

## Suggested first optimizations
1. precompiled schema indexes
2. incremental logic recomputation
3. render current page only
